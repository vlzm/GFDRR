Environment: история одной симуляции¶

Storytelling-гайд по docs/design/environment_design.md

Зачем нужен Environment¶

Представь, что у тебя есть сеть станций велошеринга. На каждой станции — какое-то количество велосипедов, между станциями ездят грузовики-ребалансировщики. Люди берут и возвращают велосипеды, грузовики перевозят излишки туда, где дефицит. Это и есть тот мир, который симулирует Environment.

Optimizer (другой потребитель модели) видит все периоды сразу и находит идеальный план. Но идеальный план — это теория. В реальности грузовик, отправленный из Depot A, недоступен, пока не доедет. В реальности спрос может быть не таким, как прогнозировали. Environment — это digital twin: он шагает по времени, принимает решения на каждом шаге исходя из текущего состояния, и показывает, что произойдёт.

Что нужно для старта¶

Чтобы запустить симуляцию, нужны две вещи:

ResolvedModelData — полное описание мира. Какие есть станции и депо, какие маршруты между ними, сколько велосипедов на каждой точке в начале, сколько грузовиков, какой спрос ожидается в каждом периоде. Всё это уже прошло через build pipeline и готово к употреблению.

EnvironmentConfig — инструкция “что делать в каждом периоде”. Упорядоченный список фаз: сначала обработай спрос, потом доставь грузы, потом прими решения о новых отправках. Порядок важен.

SimulationState: снимок мира¶

В каждый момент времени мир описывается тремя DataFrame:

inventory — сколько чего лежит на каждой станции. Таблица: facility_id x commodity_category -> quantity. На станции s1 — 12 исправных велосипедов. На депо d1 — 50.

in_transit — что сейчас в пути. Каждая запись — одна отправка: откуда, куда, что, сколько, каким ресурсом, когда отправлено, когда прибудет.

resources — где находится каждый грузовик и свободен ли он. resource_id, категория, домашняя база, текущая позиция, статус (AVAILABLE / IN_TRANSIT / BUSY / MAINTENANCE), когда станет доступен.

Плюс period_index и period_id — где мы сейчас во времени.

Неизменяемость¶

SimulationState — frozen=True. Нельзя поменять inventory “на месте”. Вместо этого — with_* методы, которые создают новый state:

new_state = state.with_inventory(updated_inventory)

Зачем? Безопасность. Если фаза получает state и что-то идёт не так — оригинальный state не повреждён. Нет сюрпризов из-за случайной мутации.

Инициализация: создаём мир¶

Когда ты создаёшь Environment(resolved, config), происходит три вещи:

  1. init_state — из данных модели собирается начальное состояние. Inventory берётся из inventory_initial. In-transit — из inventory_in_transit (маппинг дат на period_index). Resources — из resource_fleet или resources (L3). Если грузовики заданы агрегированно (“3 грузовика на депо d1”), они разворачиваются в отдельные экземпляры: rebalancing_truck_d1_0, rebalancing_truck_d1_1, rebalancing_truck_d1_2. Это векторизованная операция через index.repeat() + cumcount().

  2. SimulationLog — создаётся пустой журнал с пятью разделами.

  3. Periods — из модели извлекается список всех периодов в виде PeriodRow (NamedTuple).

Phases: что происходит в каждом периоде¶

Фаза — это единица работы внутри периода. У каждой фазы три вещи:

  • name — идентификатор (для логов)

  • should_run(period) — запускаться ли в этом периоде

  • execute(state, resolved, period) -> PhaseResult — сделать работу

PhaseResult — это пакет из четырёх вещей: обновлённый state + flow_events (что двигалось) + unmet_demand (чего не хватило) + rejected_dispatches (что не удалось отправить).

Schedule: когда запускаться¶

Не все фазы нужны в каждом периоде. Ребалансировка — только ночью (каждый 2-й период). Ремонт — раз в неделю. Schedule задаёт это:

  • Schedule.every() — каждый период

  • Schedule.every_n(3) — каждый третий

  • Schedule.custom(lambda p: p.period_type == "night") — произвольная логика

Три встроенные фазы¶

DemandPhase — люди забирают велосипеды. Фильтрует demand для текущего периода, мёрджит с inventory, вычисляет fulfilled = min(demand, available). Inventory уменьшается на fulfilled. Если demand > available — записывается unmet demand (дефицит).

ArrivalsPhase — грузовики прибывают. Фильтрует in_transit по arrival_period == current. Перемещённые товары добавляются к inventory на целевой станции. Ресурсы переключаются: IN_TRANSIT -> AVAILABLE, текущая позиция обновляется. Доставленные отправки удаляются из in_transit.

DispatchPhase — принимаем решения и отправляем. Это самая сложная фаза, потому что она делегирует решения Task’у.

Task: кто принимает решения¶

Task — это доменная логика. “Куда отправить грузовики?” — это вопрос, на который отвечает Task. Интерфейс минимальный:

class Task(Protocol):
    name: str
    def run(self, state, resolved, period) -> pd.DataFrame:
        """Верни DataFrame с отправками."""

Task смотрит на текущее состояние (где дефицит, где излишек, какие ресурсы свободны), принимает решения и возвращает таблицу dispatches: откуда, куда, что, сколько, каким ресурсом. Task может быть простым (NoopTask — ничего не делать), жадным (ближайший дефицит -> первый излишек) или сложным (VRP solver).

DispatchPhase берёт эти dispatches и:

  1. Auto-assign ресурсы — если Task не указал конкретный resource_id, DispatchPhase найдёт свободный грузовик на нужной станции

  2. Валидирует — пять проверок

  3. Применяет — обновляет state

DispatchPhase: валидация как защита¶

Прежде чем применить dispatch, DispatchPhase проверяет пять вещей:

  1. Arrival period — нельзя приехать в прошлое

  2. Edge existence — маршрут (source, target, modal_type) должен существовать в графе

  3. Resource availability — ресурс должен быть AVAILABLE и находиться на source

  4. Capacity — суммарный груз на ресурс не превышает его вместимость

  5. Inventory sufficiency — на source достаточно товара для всех отправок

Каждый непрошедший dispatch получает причину отказа (INVALID_ARRIVAL, INVALID_EDGE, NO_AVAILABLE_RESOURCE, OVER_CAPACITY, INSUFFICIENT_INVENTORY) и попадает в rejected_dispatches. Валидные — применяются: inventory на source уменьшается, создаётся запись в in_transit, ресурс помечается как IN_TRANSIT с обновлённой позицией и temporary available_at_period.

Auto-assign: когда Task не знает про ресурсы¶

Task может вернуть dispatch с resource_id=None — “отправь 5 велосипедов из d1 на s1, а каким грузовиком — разберись сам”. DispatchPhase ищет свободный ресурс на source, проверяет совместимость (resource -> commodity, resource -> modal_type) и назначает первый подходящий. Если свободных нет — dispatch отклоняется.

Цикл: как всё собирается вместе¶

Environment.run() — главный цикл:

while not is_done:
    period = periods[cursor]
    for phase in config.phases:
        if phase.should_run(period):
            result = phase.execute(state, resolved, period)
            state = result.state
            log.record_events(result, phase.name, period)
    log.record_period(state, period)
    state = state.advance_period(next)

На каждом шаге: берём текущий период, прогоняем все фазы по порядку, записываем в журнал, переходим к следующему периоду. State течёт через фазы как вода через трубу — каждая фаза читает и возвращает новый state.

Есть также step() (один период) и step_phase(name) (одна фаза) — для отладки и интерактивной работы в notebook.

SimulationLog: полная история¶

Журнал накапливает данные в пяти списках DataFrame (по одному на каждый период/фазу). В конце to_dataframes() конкатенирует всё в пять итоговых таблиц:

Таблица

Что содержит

inventory_log

Сколько чего лежало на каждой станции в каждом периоде

flow_log

Все перемещения: спрос, доставки, отправки (с именем фазы)

resource_log

Где был каждый грузовик и в каком статусе

unmet_demand_log

Где и когда не хватило велосипедов

rejected_dispatches_log

Какие отправки были отклонены и почему

Из этих таблиц можно строить любую аналитику: эволюция inventory, эффективность ребалансировки, узкие места в сети, service level.

Resource lifecycle: жизнь грузовика¶

Ресурс проходит через простой цикл:

AVAILABLE (на базе) -> dispatch -> IN_TRANSIT (в пути) -> arrival -> AVAILABLE (на месте назначения)

Ключевой момент: после прибытия ресурс остаётся там, куда приехал. Грузовик, доставивший велосипеды на станцию s3, теперь стоит на s3, а не на домашней базе. Если Task хочет его обратно — нужен отдельный dispatch. Это реалистично: в реальной жизни грузовик не телепортируется.

Трёхслойная архитектура: Phase -> Task -> Solver¶

Архитектура разделена на три уровня ответственности:

Phase — когда и как применить (schedule, validation, state update). DemandPhase, ArrivalsPhase, DispatchPhase.

Task — что решить (prepare данные, вызвать solver, постпроцессинг). RebalancerTask, RepairTask, GreedyTask.

Solver — чистая математика (VRP, LP, greedy algorithm). Не знает про state, phases или Environment.

Это позволяет комбинировать: один и тот же VRP solver можно использовать в разных Task’ах; один и тот же Task — с разными Schedule; новый домен — свои Task’и, но та же инфраструктура Phase.

Главная идея¶

Environment — это конвейер. Данные мира текут через фазы как через трубу: каждая фаза читает состояние, делает свою работу, возвращает новое состояние. Ничего не мутируется. Всё логируется. Фазы подключаемые — хочешь добавить ремонт велосипедов или ночной завоз? Напиши Phase, воткни в config.

Simulator не знает, что решает Task. Task не знает, когда его запускают. Solver не знает про Environment. Каждый уровень делает своё дело. Вместе — полноценный digital twin.
