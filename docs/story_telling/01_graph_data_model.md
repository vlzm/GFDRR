# Модель данных: история о логистической вселенной

*Storytelling-гайд по `docs/design/graph_data_model.md`*

---

## Зачем это всё

Представь, что тебе нужно управлять сетью велошеринга. Тысячи велосипедов, сотни станций, десятки грузовиков-ребалансировщиков, депо, ремонтные мастерские. Каждый день люди берут велосипеды на одних станциях и оставляют на других. К вечеру где-то пусто, где-то переполнено. Ночью грузовики перевозят велосипеды обратно. Сломанные велосипеды едут в ремонт, потом возвращаются.

Теперь представь, что ту же систему нужно использовать для банковских переводов. Или для нефтяной логистики. Или для розничной доставки. Структура одна и та же: **что-то** перемещается через **сеть узлов** с помощью **чего-то**. Меняются только конкретные названия.

Модель данных GBP — это попытка описать эту универсальную структуру один раз и навсегда. Чтобы не писать с нуля для каждого домена.

---

## Три столпа: Facility, Commodity, Resource

Вся модель стоит на трёх сущностях. Это как три измерения пространства — всё остальное выражается через них.

### Commodity — что перемещается

Это сам "поток" в сети. В велошеринге — велосипеды (исправные и сломанные). В финансах — валюты. В рознице — товары.

У commodity два уровня. **CommodityCategory** (L2) — тип потока. Именно по категориям индексируются спрос, предложение и пропускные способности. Это основной уровень работы. **Commodity** (L3) — конкретный экземпляр. Велосипед номер 4521. Нужен только если ты хочешь отследить путь конкретного предмета. Для большинства задач достаточно L2.

### Resource — чем перемещается

Это "транспорт" — то, что делает перемещение возможным. Грузовик-ребалансировщик, платёжный канал SWIFT, фургон доставки.

Аналогично два уровня. **ResourceCategory** (L2) — тип ресурса с базовыми характеристиками: вместимость, совместимые товары, совместимые виды транспорта. **Resource** (L3) — конкретный грузовик с GPS. Нужен для задач, где важно знать, какой именно ресурс где находится (симуляция, VRP).

### Facility — где commodity находится

Узлы сети. Станции, депо, ремонтные мастерские, банки, склады. В ранних версиях были отдельные "warehouse" и "receiver", но быстро выяснилось, что граница искусственная — велосипедная станция одновременно и хранит, и выдаёт, и принимает. Поэтому всё объединено в одну сущность **Facility**, а различия выражаются через три ортогональных измерения.

## Facility: три измерения одного объекта

У каждого Facility есть тип, операции и роли. Они отвечают за разное.

**FacilityType** — что это физически. Станция, депо, ремонтная мастерская. Определяет бизнес-логику и UI.

**Operations** — что объект умеет делать. Приём товара (RECEIVING), хранение (STORAGE), отправка (DISPATCH), погрузка/разгрузка (HANDLING), ремонт (REPAIR). У каждой операции свои стоимости и ёмкости. Станция умеет принимать, хранить и отправлять. Ремонтная мастерская — то же самое плюс REPAIR.

**Roles** — как объект ведёт себя в потоковой сети. Это термины из теории сетевых потоков:
- **SOURCE** — вводит commodity в сеть (у него есть supply)
- **SINK** — потребляет commodity (у него есть demand)
- **STORAGE** — хранит commodity во времени (у него есть inventory)
- **TRANSSHIPMENT** — просто перенаправляет, ничего не производит и не потребляет

Один объект может совмещать роли. Велосипедная станция — одновременно SOURCE (откуда берут велосипеды), SINK (куда возвращают) и STORAGE (где они хранятся). Депо — STORAGE + TRANSSHIPMENT (хранит и перераспределяет, но пользователи не приходят напрямую).

Роли выводятся из типа автоматически по дефолтной таблице, но их можно переопределить. Ключевой момент: при переносе на новый домен типы и операции будут другими, но роли останутся те же. SOURCE, SINK, STORAGE, TRANSSHIPMENT — это универсальные понятия потоковой сети.

## Время как фундаментальная ось

Без времени модель — статический снимок. С временем — живая система, где запасы переносятся между периодами:

```
inventory[t] = inventory[t-1] + inflow[t] - outflow[t]
```

### Planning Horizon и периоды

Временная ось задаётся через **PlanningHorizon** — независимую сущность со своим ID. Горизонт делится на **сегменты** с разной гранулярностью. Ближайшие дни планируем подневно (нужна точность), следующие недели — понедельно, дальний горизонт — помесячно. Это **multi-resolution planning**: один горизонт на 6 месяцев может состоять из 14 дней + 8 недель + 4 месяца.

Каждый **Period** имеет глобальный `period_index` (0, 1, 2, ...) для непрерывного carry-over inventory и свой `period_type` (day, week, month) от сегмента.

### Сырые данные vs. resolved

Входные данные хранятся с `date` (конкретные даты). При сборке модели запускается **time resolution** — маппинг дат на period_id с агрегацией. Зачем? Одни и те же данные можно использовать с разной гранулярностью (дневной план vs недельный), ETL проще (данные приходят с датами), а агрегация — отдельный тестируемый шаг.

### Спрос, предложение, начальные запасы

**Demand** — сколько commodity нужно на каждом SINK в каждом периоде. **Supply** — сколько доступно на каждом SOURCE. **Inventory Initial** — начальный запас на каждом STORAGE. **Inventory In-Transit** — что уже было отправлено до начала горизонта, но ещё не прибыло (граничное условие).

## Рёбра: как всё связано

Между facility проходят рёбра — маршруты для перемещения commodity.

### Multi-modal

Между двумя объектами может быть несколько рёбер разного типа. Автотранспорт и железная дорога — два разных "канала" с разными стоимостями и временем доставки. Поэтому PK ребра — это тройка: `source_id x target_id x modal_type`.

### Edge Rules

Вместо перечисления тысяч конкретных рёбер используются **правила**: "все депо соединены со всеми станциями по дороге". Edge builder берёт правила и матрицу расстояний, делает cross join и получает все рёбра. Для исключений — ручные пары (manual edges).

### Lead Time

Время доставки хранится в абсолютных часах. При сборке модели конвертируется в периоды. Нюанс: в multi-resolution режиме одни и те же 48 часов — это 2 дневных периода или 0 недельных. Поэтому resolved lead time — это не скаляр на ребре, а таблица `edge x period`.

### Допустимые товары на ребре

Каждое ребро несёт определённый набор commodity_categories. Грузовик ребалансировки везёт и исправные, и сломанные велосипеды. Но из ремонтной мастерской в депо — только исправные (обратный маршрут — только сломанные в ремонт). Это задаётся через таблицу `edge_commodity`.

## Multi-Commodity Flow и трансформация

Платформа использует формулировку **multi-commodity flow** (Ahuja, Magnanti, Orlin). Каждая commodity_category — отдельный тип потока в сети со своими demand, supply и ограничениями. Все категории на ребре делят общую пропускную способность (shared capacity).

### Трансформация: когда одно превращается в другое

Ремонтная мастерская принимает сломанные велосипеды и выпускает исправные. Банк конвертирует доллары в евро. Нефтеперегонный завод принимает нефть и выдаёт бензин, дизель и керосин.

Это N->M трансформация, разбитая на три таблицы: определение трансформации (facility + операция + loss_rate), входы (N commodity_categories с пропорциями) и выходы (M commodity_categories с пропорциями). Для простого случая "1 сломанный -> 1 исправный с 5% потерями" — по одной записи на вход и выход.

## Discrete параметры: nullable = LP-compatible

Реальная логистика полна дискретных ограничений: минимальный объём отправки, кратность партий, целое число рейсов. Ключевое решение: все discrete параметры **nullable**.

- `null` = ограничения нет (LP relaxation, непрерывные переменные)
- Значение задано = потребитель использует (MILP constraint)

Это позволяет одной моделью пользоваться и для LP (быстро, приблизительно) и для MILP (точно, медленно).

---

## Attribute System: параметры без hardcode

Стоимости, ёмкости, скорости — это числовые параметры с разной гранулярностью. Стоимость операции зависит от facility, типа операции, commodity и даты. Стоимость транспорта — от ребра, типа ресурса и даты.

Вместо фиксированных колонок (которые ломаются при смене домена) используется **AttributeRegistry**. Каждый атрибут регистрируется с указанием grain (на каком уровне он определён), kind (COST, CAPACITY, RATE...) и способа агрегации при time resolution. Подробнее — в storytelling по attribute system.

## Иерархия: масштабирование на большие сети

10 000 клиентов x 50 депо = слишком много рёбер для плоского графа. Иерархия решает это через группировку: Country -> Region -> City -> District.

Три иерархии (по одной на каждую ось):
- **Facility Hierarchy** — географическая/организационная
- **Commodity Hierarchy** — по продуктовым группам
- **Temporal Hierarchy** — multi-resolution через сегменты PlanningHorizon

Иерархия позволяет: агрегировать данные до нужного уровня для стратегического планирования (super-nodes), декомпозировать задачу (master + sub-problems), строить reporting с drill-down.

## L1 -> L2 -> L3: три уровня абстракции

Модель вложенная:

**L1** — абстрактный граф. Просто узлы и рёбра. Никакой доменной логики.

**L2** — домен-агностичная модель. CommodityCategory, ResourceCategory, Facility с ролями. Edge rules, temporal axis, transformations, multi-commodity flow. Это основной уровень. Переиспользуется между доменами и потребителями.

**L3** — конкретный домен. Station, Depot, Maintenance Hub, REBALANCING_TRUCK, велосипед #4521. Реализуется через enum'ы и конфигурацию поверх L2.

## Build Pipeline: от сырых данных к готовой модели

Сырые данные (с датами, абсолютными часами) проходят через конвейер:

1. **Validation** — проверка целостности (ссылочная, единицы, покрытие, граф, ресурсы)
2. **Time Resolution** — date -> period_id, агрегация
3. **Edge Building** — из правил и ручных пар
4. **Lead Time Resolution** — часы -> периоды (per edge x period)
5. **Transformation Resolution** — развёртка N->M
6. **Fleet Capacity** — count x base_capacity
7. **Spine Assembly** — attribute DataFrames для потребителей

На выходе — **ResolvedModelData**. Единый готовый объект для всех потребителей.

## Три потребителя: одна модель, три подхода

Самое красивое в архитектуре — одни и те же данные используются тремя принципиально разными способами:

### Optimizer (Consumer A)
Видит **все периоды сразу**. Строит LP/MILP, находит оптимальное распределение потоков на весь горизонт. Ресурсы — это capacity constraint ("из Depot A можно отгрузить X за период"), без позиций. Быстро, глобально оптимально, но "идеальный мир".

### Simulator (Consumer B)
Шагает **по периодам последовательно**. На каждом шаге: прибытия, спрос, решения, отправки. Ресурсы — с состоянием (позиция, статус, доступность). Грузовик, уехавший из Depot A, недоступен до возвращения. Медленнее, но реалистичнее.

### Analytics (Consumer C)
Читает output от Optimizer или Simulator, сравнивает с историческими данными. Plan vs Fact на одном и том же grain: edge x commodity x period_id.

Вся эта архитектура работает благодаря тому, что `ResolvedModelData` — единый контракт. Модель данных не знает про конкретных потребителей, а потребители не знают друг о друге. Каждый берёт из модели то, что ему нужно.
